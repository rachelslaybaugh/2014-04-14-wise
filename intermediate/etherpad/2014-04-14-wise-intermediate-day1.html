shell shortcuts:

- '.' means 'here', the current working directory
- '..' means 'up', the parent directory: `cd ..`
- '*' means 'everything': `ls *` or `ls *.md`
- '?' means a single character

find
find all files in a location

'find molecules -name a*.pdb'
(finds all files starting with 'a' and ending with '.pdb' in the folder molecules)
(mine worked with the command "find molecules/a*.pdb", from the shell_sandbox directory)

The gist of find:

- `find path` searches 'path'. If you don't specify anything else, it will find all files and directories in 'path'
- add `-name` or `-type` or more arguments to filter what should be matched, e.g.:
- `find . -name '*.pdb'` will show you all files that end with .pdb in the current directory
- `find molecules/ -type f` will show you every file (but not directories) in molecules/

Find can do all kinds of filters, such as by date, etc. More info on find: https://www.gnu.org/software/findutils/manual/html_mono/find.html#Finding-Files

if you want to search for files recursively starting from a directory you can do 
find . -name "a*.pdb"

grep searches the contents of files for patterns
grep ATOM *.pdb
(prints all occurences of the word ATOM in files *.pdb)

tail 
(display the end of text files)
e.g. in 2014-04-14-wise/intermediate/shell/shell_sandbox/molecules/
'tail vitamin-a.pdb'

wc
(counts the lines, words, and characters of files)

ls *.md | wc

Pipe ('|')
the character '|' routes the output of the first command to the next command
the output of the command to the left of the pipe ('|') gets sent as input of the command to the right of the pipe


Find more information on a command:

- `man [command]` get the manual page (may not work on Windows), e.g. `man wc`
- `[command] --help` often works where `man` doesn't, e.g. `ls --help`

git

intermediate/git/01-gitlocal.md

get out of the wise repository, and make a new working directory, e.g.:
cd ~/swcarpentry
mkdir git
cd git



configure git:

Tell git who you are:

    git config --global user.name "yourname"
    git config --global user.email "your@email.com"

You can see your new git config with: cat ~/.gitconfig

Tell git to color output:

    git config --global color.ui "auto"

To select an editor to use with git:

If you have no preference, nano is probably easiest:

on non-Windows, nano is easiest:

    git config --global core.editor "nano"

Note for Windows users:

on Windows, you can use notepad (this should always work, so use this if you can't get anything else to work):

    git config --global core.editor "notepad"

If you have notepad++, you can use it with:

    git config --global core.editor "C:\Program Files (x86)\Notepad++\notepad++"


To start tracking your work, you can make the current working directory a git repo:

    mkdir planets
    cd planets
    git init

Check the status:

    git status
Create a file in 'planets' called mars.txt with your editor of choice (nano, text wrangler, notepad++)

e.g. `notepad mars.txt` on Windows, or `nano mars.txt` everywhere else

Create a commit with:

    git add mars.txt
    git commit -m "Starting to think about mars"

Now modify mars.txt again, adding some more text. To see what we changed, use

    git diff

Additions will be marked with '+', deletions with '-'.

Don't forget to tell git which changes you want to commit with `git add` before `git commit`.

    `git diff` shows *un staged* changes
    `git diff --staged` shows the staged changes you are about to commit

By default `git diff` compares with the latest commit. You can compare with a different reference point by specifying its commit hash (the long hexadecimal string on the first line of the commit in `git log`:

My git log:

    commit 3670f129f060b13a3e67abb1c85e955cdc19e958
    Author: MinRK <benjaminrk@gmail.com>
    Date:   Mon Apr 14 11:43:40 2014 -0700

        create mars.txt

Compare changes in mars.txt with that commit in particular:

    git diff 3670f12 mars.txt

Undo changes with `git reset`. 

- `git reset --hard` will discard all uncommitted changes
- just `git reset` will just un-stage things, leaving all of the changes unstaged (lets you start building your commit again)

You can get an old version of a single file with `checkout`. To get mars from the first commit above (your hash will be different):

    git checkout 3670f12 mars.txt

To discard uncommitted changes of a single file, checkout HEAD:

    git checkout HEAD mars.txt

for example:

(after)     ...\;C:\AnacondaPython;C:\AnacondaPython\Scripts;C:\Program Files\Notepad++\

Windows can handle spaces in the paths; Linux/Unix doesn't like this, these files and directories will have to be double quoted on those OSs ( " " ).  Don't really matter for your local path, but, if you're going to share via GitHub or other version control sysstem with *nix users, avoid spaces in your naming structure

Collabroating with git

Create a github account on https://github.com.
Create a new repository at https://github.com/new
Link your local repository with the one on Github with:

    git remote add origin https://github.com/yourgithubuser/planets.git

And push your changes to GitHub:

    git push -u origin master
    
Note: the `-u origin master` tells your branch to 'track' the remote branch origin/master. This means you can do just `git push` in the future, without the extra arguments.

Cloning remote repos

cd to a new location, e.g. ~/swcarpentry/alternate_planets

    mkdir ~/swcarpentry/alternate_planets
    cd  ~/swcarpentry/alternate_planets

clone your repo, so you have another local copy:

    git clone https://github.com/yourgithubuser/planets.git
    cd planets
